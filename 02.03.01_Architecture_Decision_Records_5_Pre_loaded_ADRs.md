# ADR-2025-001: AI Platform Standardization — GCP Primary, AWS Secondary

**Status:** Accepted
**Date:** 2025-01-10
**Decider:** Enterprise Architect
**Consulted:** Cloud Platform Architects (GCP, AWS), CISO, CTO

## Context
Enterprise AI projects are currently selecting cloud platforms inconsistently,
leading to duplication, inconsistent security posture, increased cost,
and inability to leverage shared AI services across business units.

## Decision
- **Primary platform:** Google Cloud (Vertex AI, Gemini family)
  - Rationale: Best-in-class GenAI/LLM integration, strong unified data-to-AI
    story via BigQuery, Dataplex data governance alignment
  - CA establishes GCP as default for all new AI workloads
- **Secondary platform:** AWS (Bedrock, SageMaker)
  - Rationale: Existing workloads on AWS, Bedrock multi-model flexibility,
    strong enterprise integrations
  - Cloud Platform Architect (AWS) to maintain parity for AWS workloads
- Hybrid AI permissible where justified by ADR

## Consequences
- ✅ Consistent AI toolchain, reduced training overhead
- ✅ Centralized governance via Vertex AI Platform
- ⚠️ Migration needed for some existing AWS-only AI projects
- ⚠️ Need for cross-cloud data architecture patterns

## Compliance
- **TOGAF Phase Affected:** Phase D (Technology Architecture), Phase E (Opportunities & Solutions)
- **Guardrails:** G-001, G-003, AG-002

# ADR-2025-002: RAG Architecture Pattern for Enterprise Knowledge Management

**Status:** Proposed
**Date:** 2025-01-15
**Decider:** Enterprise Architect

## Context
12+ business units independently building RAG solutions. No shared framework,
inconsistent vector database choices, security gaps in document ingestion.

## Decision
Standardize on the following RAG pattern:
- **Ingestion:** Dataplex for data quality → Vertex AI Search (Enterprise
  index) OR OpenSearch Serverless (AWS)
- **Embedding:** textembedding-gecko@003 (GCP) / Cohere Embed v3 (AWS)
- **Generation:** Gemini 1.5 Pro (GCP) / Claude 3.5 Sonnet (AWS via Bedrock)
- **Guardrails:** Model Armor pre/post filtering
- **Evaluation:** Vertex AI Evaluation Service / custom BLEU/ROUGE pipeline

## Consequences
- ✅ Reusable RAG blueprint across all BUs
- ✅ Consistent security (VPC-SC, CMEK, IAM)
- ⚠️ Teams previously using different patterns need migration
- ⚠️ Licensing costs for third-party embedding models

## References
- PAT-001, PAT-002 (Architecture Pattern Catalog)
- POL-AI-002 (Data Governance)

# ADR-2025-003: MLOps Pipeline Standardization Using Vertex AI Pipelines (Kubeflow)

**Status:** Accepted
**Date:** 2025-01-20

## Context
ML pipelines inconsistently implemented using Airflow, custom scripts, and
notebooks. No reproducibility, no lineage tracking, difficult auditing.

## Decision
- All new ML pipelines use **Vertex AI Pipelines** (Kubeflow SDK) on GCP
- All existing pipelines migrated by Q3 2025
- Pipeline code stored in Artifact Registry versioned containers
- Pipeline templates provided as Terraform modules
- ML metadata tracked via Vertex AI Metadata Store

## Consequences
- ✅ Reproducible, auditable ML workflow
- ✅ Automated experiment tracking
- ⚠️ Kubeflow learning curve for some teams

# ADR-2025-004: Enterprise GenAI Security and Responsible AI Architecture

**Status:** Accepted
**Date:** 2025-02-01

## Context
Prompt injection, data leakage via LLM APIs, and lack of output filtering
pose risks. Multiple teams accessing GenAI without consistent controls.

## Decision
- **Prompt Security:** Model Armor (GCP) / AWS Bedrock Guardrails
  for both input and output filtering
- **Network Isolation:** All GenAI API access through Private Service Connect
  (GCP) or VPC Endpoints (AWS). No public internet egress for AI
- **DLP Integration:** All prompts scanned via DLP before submission
  (Google Cloud DLP / AWS Comprehend)
- **Content Filtering:** Block categories: PII, code, medical, legal
  (adjustable per BU)
- **Audit Logging:** All GenAI API calls logged to Cloud Audit Logs /
  CloudTrail with 2-year retention
- **RBAC:** IAM conditions restrict GenAI permissions based on user group
  AND data sensitivity label

## Compliance
- **Standards:** NIST AI RMF, ISO/IEC 42001:2023, EU AI Act
- **Guardrails:** G-004, G-005, POL-AI-007

# ADR-2025-005: Standardized LLM Evaluation and Benchmarking Framework

**Status:** Proposed
**Date:** 2025-02-10

## Context
Teams evaluating LLM responses subjectively. No consistent benchmarking,
making model comparison and production readiness decisions unreliable.

## Decision
Adopt structured evaluation framework:
1. **Benchmark Suite:** MMLU, HellaSwag, TruthfulQA + 3 enterprise-specific
   test sets per domain
2. **Automated Eval:** Vertex AI Evaluation Service (GCP) / custom
   eval harness on SageMaker (AWS)
3. **Human Eval Panel:** 5-person BU panel scores 100 production-quality
   responses per model candidate
4. **Production Canary:** 5% traffic routed to new model for 7 days
   before promotion
5. **Scorecards maintained** in Architecture Repository per model/version

## Compliance
- POL-AI-008 (ADRs), G-002 (Model Transparency)
