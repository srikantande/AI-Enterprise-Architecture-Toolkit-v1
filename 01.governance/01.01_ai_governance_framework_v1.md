## 1. Executive Summary

This document defines the AI Governance Framework for the enterprise AI portfolio, ensuring alignment with business objectives, regulatory requirements, and architectural standards as defined in our TOGAF-based Architecture Development Methodology.

### 1.1 Purpose

- Establish guardrails for AI project initiation, evaluation, approval, and lifecycle management
- Define roles, responsibilities, and decision rights across the AI portfolio
- Ensure compliance with Google Cloud recommended practices, AWS Well-Architected AI Lens, and enterprise security standards

### 1.2 Scope

This governance framework covers:

- Generative AI (Vertex AI, Bedrock, Gemini, PaLM)
- Machine Learning (AutoML, BigQuery ML, SageMaker)
- AI-powered Data Pipelines
- AI-enabled Enterprise Applications
- Conversational AI / Agents

## 4. AI Architecture Guardrails

### 4.1 Mandatory Guardrails *(Non-Negotiable)*

#### G-001: Data Sovereignty

- All enterprise AI workloads must process PII/PHI within approved geographic regions
- Cross-border data transfer requires Legal approval and DPA
- Vertex AI / Bedrock data residency settings must be explicitly configured

#### G-002: Model Transparency

- All production AI decisions affecting humans must be explainable (SHAP, LIME, or native explainability)
- Black-box models require explicit risk acceptance from C-Level
- Model Cards must be maintained per model/version

#### G-003: Human-in-the-Loop (HITL)

- AI systems making HR, financial, or clinical decisions **MUST** have human override
- Confidence thresholds must be defined and monitored
- Fallback procedures documented in runbook

#### G-004: Security & Access

- AI endpoints must use VPC Service Controls (GCP) or VPC endpoints (AWS)
- IAM policies follow least privilege — no wildcard permissions in AI roles
- Artifact Registry / ECR access restricted to CI/CD pipelines only
- API keys for AI services must be rotated every 90 days via Secret Manager

#### G-005: Ethical AI

- No AI system shall be used for automated weapons, mass surveillance, or discriminatory profiling *(per Google AI Principles & EU AI Act)*
- Bias testing required before production deployment (Fairness Indicators, What-If Tool)
- Annual AI ethics review mandatory

---

### 4.2 Advisory Guardrails *(Should Follow)*

#### AG-001: Cost Optimization

- Right-size AI/ML infrastructure using committed use discounts/CUDs
- Monitor token usage for GenAI APIs via Cloud Monitoring / CloudWatch
- Use Spot/Preemptible VMs for training workloads

#### AG-002: Reusability

- Prefer enterprise shared ML model registry over custom builds per project
- Document all use cases in Architecture Repository (TOGAF ABB)
- Leverage pre-built solutions from Google Marketplace / AWS Marketplace

#### AG-003: Observability

- All AI pipelines must emit structured logs and custom metrics
- Model drift detection must be automated (Vertex AI Model Monitoring)
- Alerting on latency, error rate, and cost anomalies required

## 6. AI Lifecycle Governance

### 6.1 Phase Gate Process

| Gate | Activities | EA Review Point |
|---|---|---|
| **G1: Ideation** | Business case, use-case scoring | Architecture viability |
| **G2: Feasibility** | PoC design, data assessment | Technical architecture review |
| **G3: Development** | Model training, pipeline build | Conformance to target architecture |
| **G4: Pre-Prod** | Security scan, bias test, penetration test | Architecture compliance sign-off |
| **G5: Production** | Deployment, monitoring, runbook | Operational readiness review |
| **G6: Retire** | Model deprecation, data archival | Decommissioning approval |

### 6.2 Model Risk Classification

| Tier | Description | Approval Required |
|---|---|---|
| 🟢 **Tier 1 — Low Risk** | Internal, non-personal, advisory | Project Lead |
| 🟡 **Tier 2 — Medium Risk** | Customer-facing, aggregated data | EA + Security |
| 🔴 **Tier 3 — High Risk** | PII/PHI, automated decisions | EA + Legal + C-Level |

## 8. AI Architecture Review Checklist

Use this checklist during every AI project architecture review:

### Pre-Architecture

- [ ] Business case approved with measurable OKRs
- [ ] Use case scored (reversibility, privacy, safety, fairness)
- [ ] Data assessment completed (availability, quality, sensitivity)
- [ ] Target state architecture defined (TOGAF Phase C/D)

### Security

- [ ] Data encryption at rest (CMEK) and in transit (TLS 1.3)
- [ ] VPC Service Controls / Endpoints configured
- [ ] IAM roles follow least privilege with audit logging
- [ ] Secret management via Secret Manager / Secrets Manager
- [ ] DLP scan on training data and prompts

### Architecture Quality

- [ ] DR strategy defined (RPO/RTO for AI workloads)
- [ ] Cost model created and approved
- [ ] Integration patterns documented (event-driven, API-led)
- [ ] Observability strategy (metrics, traces, logs, dashboards)
- [ ] Model versioning and lineage tracking enabled

### Compliance

- [ ] Regulatory impact assessment (GDPR, EU AI Act, industry-specific)
- [ ] Model Card completed
- [ ] Bias/fairness tests executed and documented
- [ ] Human-in-the-loop controls deployed *(if Tier 3)*
